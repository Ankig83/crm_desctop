# Инструкции для ИИ-агента: CRM_DESCTOP

## Обязательная дисциплина

**Перед ответом на каждый новый запрос пользователя** перечитай этот файл (`AGENTS.md`) и при необходимости `docs/TZ_ANALYSIS_AND_IMPLEMENTATION_PLAN.md`, чтобы не расходиться с зафиксированными решениями и планом.

Используй этот файл как **обязательный контекст** при любых изменениях в проекте.

## Назначение проекта

Десктоп-приложение для учёта **клиентов** и **товаров**, расчёта стоимости с учётом **акции не более одной на товар** (скидка **в процентах** либо **нет скидки**), формирования расчётных документов и отправки результатов на **e-mail**. Обмен данными с внешним миром — через **Excel** (`clients.xlsx`, `products.xlsx`, `promotions.xlsx`).

## Источник требований

Полный разбор ТЗ, архитектура и поэтапный план: `docs/TZ_ANALYSIS_AND_IMPLEMENTATION_PLAN.md`. Актуальный **план работ и статусы задач**: `docs/WORK_PLAN.md`.

## Принятые ориентиры

- **Стек:** Python 3.11+, **PySide6** (GUI), **SQLite** (основное хранилище), **openpyxl** или **pandas** (Excel), **smtplib** (почта), **PyInstaller** (сборка под Windows).

- **Объём работ:** реализуем **ровно то, что в ТЗ**, без навешивания обязательных полей, которых нет в договорённом минимуме. **Масштабирование:** схема данных и слой сервисов допускают позже добавить период акций, новые типы скидок, отчёты — без ломки ядра (nullable-поля, миграции, отдельные сервисы).

### Акции (зафиксировано заказчиком)

- На **один товар** — **не больше одной** записи акции (связь **1:0..1**).
- Скидка задаётся **только в процентах**; «нет скидки» — **0%** или **нет строки акции** на товар.
- **Период действия акции** — **обязателен по ТЗ** для каждой строки акции: без корректного периода запись не считается валидной (импорт/UI — ошибка валидации).
- **Формат дат:** **день.месяц.год** — **ДД.ММ.ГГГГ** (пример: `29.03.2026`). В UI и в Excel — этот формат; в SQLite хранить как дату (или ISO после парсинга), отображать пользователю **ДД.ММ.ГГГГ**.
- Расчёт строки: скидка **только если** дата расчёта (или «сегодня» по умолчанию) **включена** в `[дата_начала; дата_конца]` и `p > 0`; иначе полная цена.

### Сохранение данных (зафиксировано заказчиком)

- После **каждого ввода/изменения** данные **сразу** попадают в хранилище (SQLite): коммит по событию потери фокуса, по `editingFinished` или эквиваленту — без отдельной кнопки «Сохранить» как обязательного шага для основных форм. Импорт из Excel — атомарная запись пакетом по завершении импорта.

### Клиенты (решение при отсутствии детализации у заказчика)

- **Один клиент = одна запись** (одна строка в таблице / в Excel).
- Поля **Контакты**, **Адреса**, **Пункты разгрузки** — **одно многострочное текстовое поле** каждое: несколько телефонов/адресов/пунктов **с новой строки** внутри поля (удобно и для Excel-ячейки с переносами).
- **ИНН:** одна строка; валидация **10 или 12 цифр** (РФ).
- Флаг **«новый клиент»** (`is_new`) — по ТЗ; можно снять вручную после обработки.

## Поля данных (из ТЗ + уточнения)

**clients.xlsx:** ID клиента, Название, ИНН, Контакты, Адреса, Пункты разгрузки; в приложении — флаг **«новый клиент»**.

**products.xlsx:** ID товара, Наименование, Базовая цена.

**promotions.xlsx (по ТЗ):** ID товара, тип акции, размер скидки, период действия. Для строки акции **обязательны:** ID товара, размер скидки (%), **дата начала** и **дата окончания** периода в формате **ДД.ММ.ГГГГ** (рекомендуемые заголовки колонок, например: «Дата начала», «Дата окончания»). Если в шаблоне одна колонка «Период действия» — парсер должен поддерживать согласованный вид (например два числа через «—» в том же формате); **предпочтительно две колонки** для однозначности. Тип акции — по ТЗ; в логике цены используется процент.

## Правила работы в коде

1. Не хранить пароли SMTP и секреты в репозитории — только примеры в `.env.example`.
2. Импорт/экспорт Excel: валидировать заголовки и типы; при ошибках — понятный отчёт по номерам строк.
3. Запись на диск: по возможности **атомарно** (временный файл → переименование).
4. UI: валидация (ИНН, даты, числа), подсказки (tooltips), сообщения об ошибках простым языком; **немедленное сохранение** полей в БД.
5. История операций и резервное копирование — обязательные части ТЗ; не удалять без замены.
