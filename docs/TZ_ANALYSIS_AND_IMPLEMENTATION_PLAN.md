# Анализ ТЗ и план реализации: учёт товаров и клиентов

**Заказчик / исполнитель (из ТЗ):** PyBotStudio  
**Версия документа:** 1.3  
**Дата:** 29.03.2026  

### Дополнения к ТЗ (согласовано с заказчиком)

- **Акции:** на товар не более **одной** записи; скидка только **в процентах**, либо **нет скидки** (0% или нет строки акции).
- **Период действия акции:** **обязателен** (пункт структуры данных в исходном ТЗ). Для каждой строки акции задаются **дата начала** и **дата окончания**; расчёт учитывает скидку только если **дата расчёта** попадает в период **включительно** (границы уточнить в коде единообразно: обычно `[from; to]` включительно).
- **Формат дат:** **день.месяц.год** — **ДД.ММ.ГГГГ** (пример: `29.03.2026`) в Excel при импорте/экспорте и в полях UI для периода акции. Внутри БД — тип `DATE` или ISO после парсинга.
- **Подход:** делать **по ТЗ** по функциям и файлам, архитектура — с **возможностью масштабирования** (миграции, изолированные сервисы расчёта/импорта).
- **Обновление данных:** сохранение в хранилище **сразу после ввода** (без обязательной отдельной кнопки «Сохранить» для основных форм).
- **Клиенты:** принятая модель — см. §2.1 (многострочные поля, один клиент = одна запись).

*ИИ-агент: перед каждым запросом ориентироваться на актуальный `AGENTS.md`.*

## 1. Резюме

Требуется **десктоп-приложение** для менеджеров по продажам: ведение клиентов и каталога товаров, учёт **не более одной акции на товар** (скидка **в процентах**, **период действия**, даты **ДД.ММ.ГГГГ**) или отсутствие скидки вне периода/без строки акции, автоматический расчёт итоговой стоимости, формирование расчётных документов и отправка результатов **на e-mail**. Данные хранятся и обмениваются через **Excel** (`clients.xlsx`, `products.xlsx`, `promotions.xlsx`), с дополнительными требованиями к **резервному копированию**, **истории операций**, **валидации** и **подсказкам** в интерфейсе. Ввод в интерфейсе **сразу** отражается в базе данных. Реализация **в рамках ТЗ** с архитектурным **заделом на расширение**.

## 2. Разбор функциональных требований

### 2.1 Модуль клиентов

| Требование ТЗ | Интерпретация для реализации |
|---------------|------------------------------|
| База существующих клиентов | CRUD: просмотр, поиск, редактирование, удаление (с осторожностью) |
| Новый клиент с отметкой «новый» | Поле статуса/флага `is_new` или аналог; возможность снять отметку после обработки |
| Название, ИНН, телефоны, адреса, пункты разгрузки | **Решение:** одна запись на клиента; поля «Контакты», «Адреса», «Пункты разгрузки» — **текст с переносами строк** (несколько значений — каждое с новой строки). ИНН — одна строка, валидация 10/12 цифр (РФ) |

### 2.2 Модуль товаров и акций

| Требование ТЗ | Интерпретация |
|---------------|---------------|
| Загрузка каталога из Excel | Импорт `products.xlsx` с валидацией колонок и типов |
| Акционные предложения | Таблица/лист акций, связь с товаром по `ID товара`; **не более одной строки акции на товар** (уникальность `product_id`) |
| Скидка | Только **в процентах**; **нет скидки** — 0%, нет строки акции на товар или **дата расчёта вне периода** |
| Период действия | **Обязателен** для строки акции; ввод/импорт в формате **ДД.ММ.ГГГГ** (дата начала и дата окончания) |
| Автоматический расчёт стоимости | Если есть акция, `p > 0` и `valid_from ≤ дата_расчёта ≤ valid_to` (инклюзивно): `базовая_цена × количество × (1 − p/100)`; иначе без скидки |

### 2.3 Расчёт и документы

| Требование ТЗ | Интерпретация |
|---------------|---------------|
| Общая стоимость с учётом акций | Явная спецификация строк заказа/сметы + итоги |
| Расчётные документы | Экспорт: PDF и/или Excel/Word — формат не задан (нужно согласовать шаблон) |
| Отправка на e-mail | SMTP (учётные данные в конфиге, не в коде); вложения = сформированные файлы |

### 2.4 Интерфейс и надёжность

- **UI:** формы с валидацией (ИНН, числа, **даты периода акции в ДД.ММ.ГГГГ**, обязательные поля по ТЗ), tooltips/подсказки; **немедленное сохранение** изменений в SQLite после ввода (потеря фокуса / `editingFinished` и аналоги).
- **История операций:** журнал (кто/что/когда) — локально в БД или отдельный лог + экспорт.
- **Резервное копирование:** копирование файлов данных + опционально экспорт; расписание или кнопка «Создать копию».
- **Ошибки и восстановление:** транзакционная запись в SQLite при переходе с «чистого Excel»; при работе только с Excel — атомарная запись (временный файл → rename).

## 3. Открытые вопросы к заказчику (по оставшимся пунктам)

1. **Расчётный документ:** вид (счёт, КП, спецификация), шаблон, реквизиты продавца.  
2. **Email:** настройка SMTP в приложении; нужна ли отдельная история отправок в UI.  
3. **«Автоматическое обновление данных» из исходного ТЗ:** трактовать как **мгновенное обновление БД при вводе** (согласовано) **плюс** при необходимости — повторный импорт Excel вручную или по кнопке; уточнить, нужен ли опрос сетевой папки по таймеру.

**Зафиксировано:** период акции **обязателен**; формат дат **ДД.ММ.ГГГГ**; для расчёта — две даты в Excel (рекомендуется) или парсинг одной колонки «Период» по согласованному шаблону. Часовой пояс: **локальная дата ПК** для даты расчёта, если не задана явно в форме расчёта.

Закрыто ранее: несколько акций на товар (не более одной); тип скидки — только проценты; структура полей клиента — многострочный текст.

## 4. Рекомендуемая архитектура

### 4.1 Стек (ориентир: Python + десктоп)

| Слой | Рекомендация |
|------|----------------|
| UI | **PySide6 / PyQt6** — нативный вид на Windows, формы, таблицы, диалоги |
| Данные | **SQLite** как основное хранилище + **синхронизация импорт/экспорт с Excel** (openpyxl / pandas) |
| Документы | **reportlab** или **python-docx** + конвертация; либо шаблон Excel → заполнение → PDF через LibreOffice (опционально) |
| Почта | **smtplib** + конфиг `.env` / зашифрованный локальный конфиг |
| Сборка | **PyInstaller** — один `.exe` для Windows |

**Почему не «только Excel»:** одновременный доступ, история, валидация и резервы проще с БД; Excel остаётся форматом обмена с бухгалтерией/заказчиком, как в ТЗ.

**Альтернатива:** чистый Excel + VBA — хуже по тестируемости и UI; не рекомендуется при заявленных сроках и модульности.

### 4.1.1 Принципы масштабирования

- Таблицы с **nullable** полями для будущих атрибутов (не для обхода обязательного периода акции).
- **Миграции** схемы SQLite (версия `PRAGMA user_version` или скрипты при росте проекта).
- Бизнес-правила **в сервисном слое** (расчёт цены, импорт, парсинг дат **ДД.ММ.ГГГГ**), а не размазаны по виджетам UI.
- Импорт Excel: пустые или неверные даты периода — **ошибка строки** с понятным сообщением.

### 4.2 Модули приложения (схема)

```
┌─────────────────────────────────────────────────────────┐
│                      UI (главное окно)                   │
├──────────────┬──────────────┬──────────────┬────────────┤
│   Клиенты    │   Товары     │   Акции      │  Расчёт/   │
│              │   (импорт)   │              │  документы │
├──────────────┴──────────────┴──────────────┴────────────┤
│              Сервисный слой (бизнес-логика)              │
├──────────────────────────┬──────────────────────────────┤
│  Репозитории (SQLite)    │  Адаптеры Excel (импорт/экспорт) │
├──────────────────────────┴──────────────────────────────┤
│  Журнал операций │ Резервное копирование │ Почта (SMTP)   │
└─────────────────────────────────────────────────────────┘
```

### 4.3 Логическая модель данных (минимум)

- **clients:** id, name, inn, contacts, addresses, unload_points, is_new, created_at, updated_at  
- **products:** id, name, base_price, imported_at  
- **promotions:** id, product_id (**UNIQUE**), discount_percent (0…100), `promo_type` (как в ТЗ), `valid_from`, `valid_to` (**NOT NULL** для существующей строки акции; хранение как DATE)  
- **calculation_sessions / orders** (для истории): id, client_id, created_at, total, snapshot json или нормализованные строки  
- **audit_log:** timestamp, action, entity, details  

Связь акции с товаром — **не более одной записи** на `product_id` (**1:0..1**).

## 5. Детальный план реализации по этапам

### Этап 0. Подготовка (2–3 дня) — совпадает с ТЗ

1. Уточнить оставшиеся пункты раздела 3 (шаблон документа, SMTP, фоновый импорт).  
2. Спроектировать схему SQLite и согласовать **точные имена колонок** Excel с шаблонами.  
3. Создать **шаблоны** `clients.xlsx`, `products.xlsx`, `promotions.xlsx` (первая строка — заголовки).  
4. Репозиторий: структура папок, `requirements.txt`, линтер, минимальный CI (опционально).

### Этап 1. Ядро данных (≈2 дня)

1. Миграции/создание БД, репозитории для клиентов, товаров, акций.  
2. Импорт Excel → валидация → запись в БД (отчёт об ошибках по строкам).  
3. Экспорт обратно в Excel (для обмена и резерва).

### Этап 2. Бизнес-логика акций и расчёта (≈1–1.5 дня)

1. Одна акция на товар: уникальность `product_id` в БД и при импорте.  
2. Парсинг дат **ДД.ММ.ГГГГ** при импорте и из UI; `valid_from` ≤ `valid_to`.  
3. Формула строки: если акция есть, `p > 0` и дата расчёта в периоде — `базовая_цена × qty × (1 − min(p,100)/100)`; иначе без скидки.  
4. Юнит-тесты: границы периода (день начала/конца), вне периода, `p=0`, 100%, нет акции, дубль `product_id`, неверный формат даты.

### Этап 3. UI (≈3 дня)

1. Список/форма клиентов с флагом «новый», поиск, валидация ИНН, многострочные поля контактов/адресов/разгрузки; **save-on-change** в БД.  
2. Таблица товаров, кнопка «Загрузить из Excel»; правки в таблице — **сразу** в БД.  
3. Таблица/форма акций: один товар — одна строка, скидка в %, **дата начала и окончания** (ДД.ММ.ГГГГ); **сразу** в БД.  
4. Мастер/форма расчёта: выбор клиента, строки (товар, количество), **дата расчёта** (по умолчанию сегодня, ДД.ММ.ГГГГ), превью цен и итога.  
5. Подсказки (tooltips) и сообщения об ошибках.

### Этап 4. Документы и e-mail (≈1–2 дня)

1. Генерация расчётного документа по шаблону.  
2. Настройки SMTP в отдельном конфиге; отправка с вложением; запись в историю.

### Этап 5. Надёжность и история (≈1 день)

1. Резервное копирование: копия БД + последние Excel в выбранную папку.  
2. Журнал операций (импорт, расчёт, отправка письма).  
3. Обработка ошибок: пользовательские сообщения, лог файла для диагностики.

### Этап 6. Тестирование (3 дня) — по ТЗ

1. Чек-лист по критериям приёмки.  
2. Нагрузочно-малые объёмы: большие Excel (1000+ строк).  
3. Безопасность: не хранить пароль SMTP в открытом виде; ограничение путей к файлам.

### Этап 7. Внедрение (2 дня)

1. Сборка `PyInstaller`, инсталлятор (опционально Inno Setup).  
2. Руководство пользователя, описание структуры данных, установка.  
3. Короткое обучение (скринкаст или сессия).

**Итого по календарю ТЗ:** подготовка + разработка + тесты + внедрение — **реалистично при закрытых пробелах в ТЗ**; буфер 20–30% на согласования.

## 7. Риски и меры

| Риск | Мера |
|------|------|
| Конфликт импорта Excel (две строки акции на один товар) | Валидация при импорте, отчёт об ошибке; в UI — запрет дубля по `product_id` |
| Неверный формат дат в акциях | Строгий парсер **ДД.ММ.ГГГГ**, сообщение с номером строки Excel |
| Повреждение Excel при записи | Атомарная запись, бэкап перед импортом |
| SMTP блокируется провайдером | Поддержка app-password, TLS; инструкция в документации |
| Объём документации | Шаблоны документов параллельно с UI |
